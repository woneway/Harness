# Harness v2 演进设计

> 从 AI 编排框架到快速应用工厂

## 1. 重新定位

### v1 定位
Claude CLI 优先、开箱即用的 AI Workflow 框架。用户手写 Python pipeline。

### v2 定位
**程序员的 Dify** — 用自然语言描述需求，Harness 自动拆解、确认、执行。

核心理念：
- **NL 驱动**：一段话 → 可运行的任务
- **代码级灵活**：生成的是真正的 Python，可查看/修改/复用
- **插件生态**：集成新服务像装 pip 包一样快
- **双模式运行**：一次性 Pipeline + 长驻服务
- **生产就绪**：调度、重试、监控、通知内置

### 与竞品的差异化

| 维度 | Dify/n8n | LangGraph/CrewAI | Harness v2 |
|------|----------|-----------------|------------|
| 用户 | 非程序员 | 开发者 | 开发者 |
| 交互 | 可视化拖拽 | 写代码 | **自然语言 + 代码** |
| 灵活度 | 受限于组件 | 完全自由 | **NL 快速启动，代码兜底** |
| 运行 | 云端 | 库/框架 | **本地优先，无人值守** |
| 集成 | 400+ 内置 | 需自己写 | **Adapter 插件** |

## 2. 核心场景验证

### 场景一：热点话题 AI 短视频（一次性 Pipeline）

```
$ harness run "帮我做一个热点话题AI短视频"

[Harness] 正在分析你的需求...
[Harness] 检测到需要以下能力:
  - 热点数据源 → 已安装: weibo-trending
  - TTS 服务   → 已安装: minimax-tts
  - 视频生成   → 未安装!
[Harness] 需要安装 video-gen。确认？(Y/n)

[Harness] 执行计划:
  1. 获取微博热搜 Top5
  2. 选择最热话题，生成 60s 视频脚本
  3. MiniMax TTS 生成语音
  4. MiniMax Video 生成视觉
  5. ffmpeg 合成最终视频

[Harness] 确认执行？
  > 直接执行 (Y)
  > 生成项目后执行 (g)
  > 修改建议 (e)
  > 取消 (n)

$ e
$ 用抖音热点而不是微博，时长改30秒，加字幕

[Harness] 已调整计划:
  1. 获取抖音热点 Top5               ← 已修改
  2. 选择最热话题，生成 30s 视频脚本  ← 已修改
  3. MiniMax TTS 生成语音
  4. MiniMax Video 生成视觉
  5. ffmpeg 合成最终视频（含字幕）     ← 已修改

[Harness] 确认执行？(Y/g/e/n)
$ Y

[Harness] 执行中...（全自动，失败重试，完成通知）
```

**特征**：一次性运行，跑完结束，输出文件。

### 场景二：股票讨论群（长驻服务 + 事件驱动）

```
$ harness run "建一个A股讨论群，4个角色，盘中异动和收盘都要讨论"

[Harness] 执行计划:
  角色:
    - 技术分析师: 看K线、均线、MACD
    - 基本面研究员: 看财报、行业趋势
    - 短线交易员: 关注资金流、龙虎榜
    - 风控经理: 提示风险、控制仓位
  触发:
    - 定时: 每天 15:05（收盘后）
    - 事件: 个股涨跌超 3%、龙虎榜更新
  每轮讨论:
    1. 获取最新行情数据
    2. 4角色讨论（10轮）
    3. 输出观点报告
  记忆:
    - 保留历史讨论，角色可回顾之前的判断

  > 确认 (Y) | 修改建议 (e) | 生成项目 (g) | 取消 (n)
```

**特征**：长期运行，定时 + 事件双触发，角色有长期记忆。

### 两个场景对比揭示的架构需求

| 能力 | 短视频 | 讨论群 | v1 状态 |

| NL → 执行计划 | 需要 | 需要 | 无 |
| 用户确认/修改建议 | 需要 | 需要 | 无 |
| Adapter 插件生态 | 需要 | 需要 | 无 |
| 一次性 Pipeline | 核心 | - | 已有 |
| 长驻服务 + 事件触发 | - | 核心 | 部分 |
| 长期记忆 / 角色状态 | - | 核心 | 部分 |

## 3. 架构设计

### 三层架构

```
┌─────────────────────────────────────────────┐
│  Layer 3: NL Planner（大脑）                  │
│                                             │
│  自然语言 → 意图识别 → 任务拆解 → 执行计划     │
│  复杂任务 → 与用户确认/接受修改建议 → 再执行    │
│  了解所有已安装的 Adapter，选择合适的组合       │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Layer 2: Integration Registry（生态）        │
│                                             │
│  harness add minimax-tts                    │
│  harness add stock-data                     │
│  harness add weibo-trending                 │
│                                             │
│  每个 Adapter:                               │
│    - 标准接口（submit / poll / query）        │
│    - 配置模板（API key、endpoint）            │
│    - 元信息（描述、能力标签 → Planner 选择用） │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Layer 1: Core Engine（底座）                 │
│                                             │
│  现有能力（保留）:                            │
│    Task 类型 | Runner | Storage | Memory     │
│    Scheduler | Notifier | Parallel | Dialogue│
│                                             │
│  新增能力:                                   │
│    - Claude Agent SDK 替换 CLI 子进程         │
│    - 长驻服务模式（daemon + 事件触发）         │
│    - 条件分支 / 简单循环                      │
│    - 角色长期记忆                             │
└─────────────────────────────────────────────┘
```

### 交互模型

所有场景统一的用户交互流程：

```
用户输入（自然语言）
    │
    ▼
NL Planner 分析
    │
    ├─ 检查 Adapter 可用性（缺少则提示安装）
    │
    ▼
展示执行计划
    │
    ├─ Y: 直接执行
    ├─ g: 生成项目代码后执行（可查看/修改/复用）
    ├─ e: 用户输入修改建议 → 重新生成计划
    └─ n: 取消
    │
    ▼
执行（全自动）
    │
    ├─ 一次性: 跑完输出结果
    └─ 长驻: 作为后台服务持续运行
```

### 执行模式

| 模式 | 触发 | 生命周期 | 示例 |
|------|------|---------|------|
| **Pipeline** | 手动 / cron | 跑完结束 | 短视频生成、代码分析 |
| **Service** | 事件 + 定时 | 持续运行 | 讨论群、监控告警 |

两种模式共享底层 Task/Runner/Storage/Memory，区别在于执行引擎。

## 4. 竞品分析

### v2 需要借鉴的设计

| 框架 | Stars | 借鉴什么 | 不借鉴什么 |
|------|-------|---------|-----------|
| **Claude Agent SDK** | 新 | 替换 CLI 子进程，获得原生 tool use | - |
| **LangGraph** | 24.8K | 条件边、循环的概念 | 完整图状态机（过重） |
| **CrewAI** | 45.9K | Agent 角色抽象、记忆系统 | Manager 层级（不需要） |
| **Pydantic AI** | 15.2K | 类型安全、依赖注入 | 重新造 Agent 抽象 |
| **OpenAI Agents SDK** | 30K+ | Handoff 优雅的控制权转移 | OpenAI 绑定 |
| **Dify** | 134K | 400+ 集成生态的组织方式 | 可视化拖拽 |

### Harness 独特价值（v2 保持）

1. **本地优先、纯 Python**：不需要服务器/容器/云
2. **混合任务类型**：LLM + Function + Shell + Polling 统一编排
3. **无人值守**：cron 调度 + 重试 + 通知，真正的自动化
4. **代码级灵活**：NL 快速启动，但可随时降级到 Python 精细控制

### Harness v2 新增价值

5. **NL 驱动**：自然语言描述 → 执行计划 → 确认 → 运行
6. **插件生态**：`harness add xxx` 快速集成新服务
7. **双模式运行**：一次性 Pipeline + 长驻 Service
8. **用户参与**：确认/修改建议的人机交互循环

## 6. 设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 用户入口 | 命令行 NL（`harness run "..."`） | 程序员友好，不需要 Web UI |
| 执行确认 | 四选项（Y/g/e/n） | 灵活：直接跑、生成项目、修改、取消 |
| 修改建议 | 自然语言输入 | 最自然的交互方式 |
| 执行过程 | 全自动，不中断 | 无人值守是核心价值 |
| Adapter 来源 | 手写预置（方案A） | 比 AI 自动生成更可靠 |
| 架构风格 | 自建核心 + 可选接入专业框架 | B+C 混合：掌握核心，不重复造轮子 |
| CLI 替换 | Claude Agent SDK | CLI 子进程太原始，SDK 是官方升级路径 |
